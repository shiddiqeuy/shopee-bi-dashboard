| Code review | Get feedback and improve your code quality |
| Industry workflow | Practice issues, branches, commits, pull requests, and reviews |
| Learning by building | Understand faster by making real changes |
| Networking | Meet other developers and contributors |

# First Contribution

If this is your first time contributing, start small. Small wins build confidence.

## Issue Difficulty Guide

| Level | Good For | Examples |
| :--- | :--- | :--- |
| 🟢 Beginner | First-time contributors | Typo, documentation, README update, small bug |
| 🟡 Intermediate | Contributors with some project understanding | Bug fix, refactor, unit testing |
| 🔴 Advanced | Contributors ready for bigger tasks | New feature, architecture change, optimization |

Recommended first steps:

1. Look for issues labeled `good first issue`, `beginner`, or `documentation`.
2. Read the issue carefully.
3. Ask questions if anything is unclear.
4. Start with one small task.

> [!IMPORTANT]
> Your first Pull Request does not need to be perfect. The goal is to learn the workflow.

---

# Before You Start

Before working on a task, please check this list:

- [ ] Read the issue carefully
- [ ] Ask questions if the task is unclear
- [ ] Comment on the issue before working on it
- [ ] Assign yourself if assignment is available
- [ ] Create a new branch
- [ ] Keep your commits small and focused
- [ ] Test your changes
- [ ] Update documentation if needed

Good branch names:

```text
fix/login-error-message
docs/update-readme-setup
feature/add-upload-validation
test/add-api-tests
```

Good commit messages:

```text
fix: correct dashboard filter label
docs: improve setup instructions
test: add unit tests for upload validation
```

# Pull Request Checklist

Before opening a Pull Request, please make sure:

- [ ] My PR has a clear title
- [ ] My PR explains what changed
- [ ] My PR is linked to an issue or discussion if available
- [ ] I kept the change focused and not too large
- [ ] I tested my changes locally if possible
- [ ] I updated documentation if needed
- [ ] I did not include secrets, passwords, API keys, or private data
- [ ] I am open to feedback and code review

PR title examples:

```text
docs: add setup guide for beginners
fix: handle empty upload state
test: add unit tests for ETL status endpoint
```

> [!NOTE]
> Code review is not personal. It is how we learn and make the project better together.

# Open Source Mindset

Open source is not about being perfect.

It is about showing up, learning, improving, and helping others.

**Progress > Perfect**

Your first PR does not need to be amazing.

Your code may need changes. That is normal.

Your idea may need discussion. That is normal too.

Every contributor was once a beginner.

Every small improvement matters.

Every question can help someone else who is also learning.

> [!TIP]
> The best contributors are not always the people who know everything. They are the people who keep learning and help others learn too.
